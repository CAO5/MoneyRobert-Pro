# MoneyRobert Pro 系统评估与演进规划

> 文档版本：1.0  
> 创建日期：2026-06-22  
> 文档类型：系统评估与战略规划

---

## 一、总体结论

MoneyRobert Pro 已具备"加密资产智能分析与交易平台"的完整产品骨架：行情、技术分析、衍生品数据、多 Agent 辩论、模拟盘、回测、自动交易、风险确认、记忆和自进化都已覆盖。

### 当前定位

**AI 辅助交易工作台 + 规则风控系统**，而不是已经统计验证的量化预测与稳定盈利系统。

### 核心差距

主要差距不在页面或功能数量，而在三个基础层：

1. **回测账本和撮合正确性尚未得到保证**
2. **置信度主要来自 Agent 主观评分，不等于经过校准的上涨概率**
3. **数据、组合资金模型与样本外验证不足以支撑"稳定走势预测"或实盘自主交易**

## 二、关键发现

### P0：回测账本存在正确性风险

**问题位置**：`backend/src/backtest/matching_engine.rs:151`

**问题描述**：
1. 完全平掉反向仓位时执行 `cash += notional + realized_pnl`，退出金额已经反映价格变化，再加一次盈亏可能造成**重复记账**
2. 反向订单数量大于现有仓位时，代码直接从原仓位减去全部订单数量，可能产生**负仓位**，却没有为剩余数量建立新方向仓位

**影响**：
- 污染权益曲线、收益率、回撤和夏普率
- 在修复并建立资金守恒测试前，**不应将当前回测结果用于模型晋级或实盘授权**

### P0：绩效指标不完整

**问题位置**：`backend/src/backtest/performance_engine.rs:21`

**当前问题**：
- 夏普率采用"每日最高权益"近似日末权益，会系统性美化表现
- Sortino 固定为 0
- 总滑点成本固定为 0
- 缺少基准收益、Alpha/Beta、Calmar、CVaR、尾部风险和回撤持续期

### P0：测试通过，但金融核心缺乏覆盖

**实际验证结果**：
- 后端：43 个测试通过
- 前端：生产构建通过
- Rust 编译约 110 条警告

**覆盖缺口**：
- 撮合引擎只有一个滑点测试，没有覆盖开仓、加仓、减仓、反手、费用与资金守恒
- `backend/tests/agents/` 下的大量测试没有被当前 Cargo 集成测试入口执行

### P1：部分用户数据隔离不完整

**问题位置**：`backend/src/routes/reports.rs:36`

**问题描述**：
虽然要求登录，但查询、更新、删除报告均未按 `user_id` 隔离。普通用户可能看到或修改全局报告。

**建议修复**：
- 增加所有权字段
- 服务层授权检查
- PostgreSQL RLS（行级安全）

## 三、多视角深度分析

### 3.1 经济学家视角

#### 当前对应的经济学思想

当前多 Agent 结构接近以下模型的混合体：

1. **异质预期模型**：不同 Agent 使用技术、资金、新闻信息形成不同预期
2. **有限理性模型**：决策依赖有限指标、规则和 LLM 解释
3. **信息聚合/贝叶斯模型平均的雏形**：基金经理按部门和置信度加权意见
4. **行为金融**：利用情绪、追涨杀跌、拥挤度和反向信号
5. **凯恩斯"选美竞赛"**：Agent 不只判断价值，也判断其他市场参与者如何判断市场

#### 当前局限

**问题位置**：`backend/src/agents/agents.rs:193`

目前把观点压缩为 `上涨=1、下跌=-1、中性=0`，再使用固定部门权重；没有效用函数、均衡条件、风险溢价或结构参数估计。

#### 建议扩充

1. **宏观流动性模型**
   - 美元指数、实际利率、M2、稳定币供应、ETF 净流量

2. **市场微观结构**
   - 订单簿不平衡、买卖冲击、流动性溢价

3. **期限结构模型**
   - 永续资金费率、期货基差、期限价差

4. **反身性与拥挤交易**
   - 价格上涨→杠杆增加→清算风险积累

5. **状态切换模型**
   - 风险偏好、避险、趋势、震荡、流动性危机五类市场状态

6. **投资者效用模型**
   - 从"最大预测准确率"转为"最大化风险调整后效用"

**现状评估**：宏观 Agent 虽已存在，但当前数据输入不足；"链上分析 Agent"也缺少真实链上数据支持，角色名称领先于数据能力。

### 3.2 基金经理视角

#### 数据完整性

**当前数据采集**（`backend/src/collector.rs:8`）：
- 20 个 USDT 永续合约
- Ticker（仅保留 24 小时）
- 六个周期 K 线
- 资金费率（仅保留 7 天）

**缺失数据**：
- 逐笔成交、主动买卖量、CVD
- L2/L3 订单簿及深度变化
- 爆仓和强平数据
- 期权隐含波动率、偏度、期限结构
- 现货—永续—交割合约基差
- 多交易所价差和资金费率
- 稳定币流入流出、交易所净流量、巨鲸地址
- 宏观及监管事件的结构化事件数据

**问题**：持仓量和多空比主要由接口请求时临时访问 OKX，并未形成稳定完整的数据管线。Ticker 仅保留 24 小时、资金费率仅 7 天，不足以训练或验证中长期模型。

#### 技术模型

**当前模型**：
SMA、EMA、RSI、MACD、布林带、ATR 及简单多周期判断为主，适合作为特征，不适合单独承担预测。

**建议扩充**：

| 类别 | 指标 |
|------|------|
| **趋势** | ADX、Donchian、SuperTrend、时间序列动量 |
| **量价** | VWAP、Anchored VWAP、OBV、MFI、CVD |
| **波动率** | 实现波动率、Parkinson/Garman-Klass、GARCH |
| **微观结构** | spread、depth imbalance、Kyle Lambda、价格冲击 |
| **统计套利** | 协整、均值回复、跨交易所和跨期限基差 |
| **市场状态识别** | HMM、Change Point、聚类 |
| **机器学习** | LightGBM/XGBoost 作为表格数据基线，而非直接以 LLM 预测价格 |

#### 资金模型

**当前模型**（`backend/src/agents/agents.rs:371`）：
```
仓位 = 最大仓位 × Agent置信度 × 波动率调整
```
置信度越高，杠杆越高。

**问题**：这不是完整的组合资金模型，因为置信度没有概率校准，容易把"模型说得自信"误认为"交易优势更大"。

**建议采用**：
1. **Fractional Kelly**：只使用经校准胜率与盈亏比
2. **波动率目标仓位**
3. **单笔风险预算**：每笔最多损失 NAV 的 0.25%–0.5%
4. **组合相关性和风险贡献限制**
5. **CVaR、最大回撤和流动性约束**
6. **资金费率、手续费、滑点、冲击成本后净期望**
7. **分层止损、时间止损和策略失效止损**

### 3.3 数学分析师视角

#### 预测目标重构

无法设计一个保证"稳定预测"的市场模型，但可以构建**可检验、可校准的条件概率和数学期望系统**。

**建议预测目标**：不要直接设为"价格是多少"，而是：

$$r_{t,h} = \ln(P_{t+h}/P_t)$$

**模型输出**：
- $P(r_{t,h} > \theta)$：上涨概率
- $P(r_{t,h} < -\theta)$：下跌概率
- 收益分位数 $Q_{10}, Q_{50}, Q_{90}$
- 预期波动率
- 最大不利变动 MAE
- 模型不确定度

#### 交易净期望

$$EV = p_{up}\mu_{up} + p_{down}\mu_{down} + p_{flat}\mu_{flat} - fee - slippage - funding - impact$$

**交易条件**：
$$EV > EV_{min}, \quad P(EV > 0) > p_{min}, \quad CVaR_{\alpha} < RiskBudget$$

#### 特征工程（六组）

1. **价格动量与均值回复**
2. **成交量、订单流和流动性**
3. **资金费率、基差、持仓量和清算**
4. **波动率与市场状态**
5. **跨资产、跨交易所相关性**
6. **新闻、宏观和链上事件**

#### 验证方法

必须采用：
- **Walk-forward** 验证
- **Purged K-fold + embargo**
- 严格防止未来函数
- **分市场状态评估**：牛市、熊市、震荡和危机
- **评估指标**：Brier Score、Log Loss、校准曲线，而不只看胜率
- **基准对比**：模型预测与"永远看涨""趋势延续"等简单基准比较

**重要原则**：LLM 更适合抽取事件和生成解释，不应作为最终价格概率的唯一来源。

### 3.4 系统架构师视角

#### 优点

1. Rust/Axum、Vue、PostgreSQL、Redis 的技术选择合理
2. 行情、交易、Agent、回测、风控已有模块边界
3. API Key 加密、JWT、紧急停止和风险确认具备产品意识
4. 前后端均可成功构建

#### 主要问题

1. **代码组织**
   - `backend/src/routes/ai_analysis.rs` 超过 4000 行，混合 HTTP、数据获取、指标、Prompt、辩论、评分和交易计划
   - 多套重复决策/回测/模拟实现，容易出现规则漂移

2. **数据管线**
   - 采集器按交易对和周期串行轮询
   - 错误大量用 `.ok()` 吞掉，缺少数据质量告警
   - 数据源、特征、模型和预测没有版本血缘

3. **扩展性**
   - 自动交易任务主要依赖进程内任务，不适合可靠恢复和水平扩容
   - 内存限流与单实例状态无法自然扩展
   - 缺少"模型服务"和"交易执行服务"之间的硬隔离

#### 推荐目标架构

保留**模块化单体**，不必立即微服务化，但应建立清晰六层：

```
Market Data → Feature Store → Model/Signal → Portfolio/Risk → Execution → Attribution
```

**关键原则**：
1. LLM/Agent 只能生成候选信号
2. 风控和订单执行必须是确定性代码
3. 每次决策保存数据版本、特征版本、模型版本和规则版本
4. 模拟盘、回测和实盘共用同一套组合与执行语义
5. 建立幂等订单、Outbox、审计日志、断路器及数据新鲜度监控
6. 使用 Decimal 表达资金，统一时区和 symbol 规范

### 3.5 产品设计经理视角

#### 核心理念

目标应是"**可验证的信任**"，而不是让用户盲目依赖系统。

#### 决策卡升级

把目前的"置信度百分比"升级为决策卡：

| 字段 | 说明 |
|------|------|
| **建议动作** | 做多/做空/观望 |
| **预测周期** | 时间范围 |
| **概率分布** | 上涨、下跌、震荡概率 |
| **预期收益** | 最乐观区间 |
| **最坏区间** | 风险边界 |
| **成本预估** | 手续费、滑点、资金费率 |
| **证据来源** | 数据更新时间 |
| **支持证据** | 看多因素 |
| **反对证据** | 看空因素 |
| **历史表现** | 历史同类场景表现 |
| **适用状态** | 模型适用状态和失效条件 |
| **仓位建议** | 建议仓位及最大可能损失 |

#### 产品主流程重构

```
发现机会 → 查看证据 → 模拟结果 → 设置风险预算 → 执行 → 事后归因
```

#### 展示优化

1. **默认展示净收益**，不展示忽略成本的毛收益
2. **胜率旁必须同时展示**：盈亏比、回撤和样本量
3. **置信度表达**：将"80% 置信度"改成"历史校准后，类似信号上涨率为 X%，样本 N"
4. **强制流程**：实盘前强制通过影子模式和模拟盘
5. **交易闭环**：每笔交易提供"为什么做、何时证明错误、为何退出"的完整闭环
6. **风险警示**：对数据过期、Agent 分歧、模型失配明确显示"暂不建议交易"

## 四、实施路线图

### 第一阶段：可信底座

**目标**：修复基础正确性问题，建立测试保障

**任务**：
1. ✅ 修复撮合与账户记账
2. ✅ 建立资金守恒、反手、费用、滑点测试
3. ✅ 修复报告数据隔离
4. ✅ 接通全部已有测试
5. ✅ 修正夏普、Sortino、滑点和权益曲线
6. ✅ 暂停依据当前回测结果自动晋级实盘

**验收标准**：
- 撮合引擎通过完整的资金守恒测试套件
- 所有绩效指标准确计算
- 用户数据完全隔离
- 测试覆盖率 > 80%

---

### 第二阶段：量化基础

**目标**：建立数据基础设施和概率预测能力

**任务**：
1. 建立长期历史数据和特征仓库
2. 增加订单簿、成交、清算、基差和跨交易所数据
3. 建立市场状态模型及概率校准
4. 使用 Walk-forward 样本外验证

**验收标准**：
- 历史数据覆盖 > 2 年
- 概率校准曲线接近对角线
- 样本外测试显著优于简单基准

---

### 第三阶段：组合与资金管理

**目标**：实现科学的资金配置和风险控制

**任务**：
1. Fractional Kelly + 波动率目标
2. 相关性、风险贡献、CVaR 和流动性约束
3. 统一模拟、回测和实盘执行模型

**验收标准**：
- 组合最大回撤可控
- 风险贡献透明可追溯
- 模拟与实盘执行结果一致

---

### 第四阶段：可信产品体验

**目标**：提供可验证、可解释的交易决策支持

**任务**：
1. 概率决策卡
2. 数据血缘和证据时间线
3. 模型卡、校准曲线、反事实解释
4. 交易后自动归因和策略失效提醒

**验收标准**：
- 每笔交易都有完整的决策证据链
- 用户可以理解并验证系统建议
- 策略失效时及时告警

---

## 五、总结

**一句话判断**：

项目的方向是对的，产品骨架也相当完整；现在最值得投入的不是继续增加 Agent，而是把"**账算对、概率校准、风险组合化、证据可追溯**"做深。
