## 一、总体结论

MoneyRobert Pro 已具备“加密资产智能分析与交易平台”的完整产品骨架：行情、技术分析、衍生品数据、多 Agent 辩论、模拟盘、回测、自动交易、风险确认、记忆和自进化均已有覆盖。

当前系统更准确的定位是：**AI 辅助交易工作台 + 确定性风控执行系统**，尚不是已经通过统计检验的量化预测与稳定盈利系统。

主要差距不在页面或 Agent 数量，而在四个基础能力：

1. 回测账本、撮合、费用、滑点和权益曲线必须持续用测试证明正确。
2. Agent 置信度必须从主观评分升级为可校准概率、净期望和历史样本表现。
3. 数据管线必须覆盖长期历史、特征版本、市场状态和数据血缘。
4. 模拟盘、回测和实盘必须共享组合、风控和执行语义。

本次核验后，原文档中的第一阶段不应再笼统标记为“全部完成”。最新代码已经修复或启动了多项关键工作，但仍存在部分设计与实现缺口。

## 三、目标设计原则

系统应保留模块化单体，不必立即微服务化，但必须建立清晰六层：

```text
Market Data -> Feature Store -> Model/Signal -> Portfolio/Risk -> Execution -> Attribution
```

| 层级 | 职责 | 不应承担的职责 |
|------|------|----------------|
| Market Data | 采集、清洗、去重、时间对齐、质量标记 | 直接生成交易建议 |
| Feature Store | 计算、存储和版本化特征 | 临时拼接不可复现特征 |
| Model/Signal | 输出概率、分位数、状态和解释 | 直接决定仓位或下单 |
| Portfolio/Risk | 组合约束、仓位预算、风险门禁 | 修改模型概率 |
| Execution | 幂等下单、撮合、订单状态机、审计 | 绕过风控执行 |
| Attribution | 绩效、归因、校准、复盘 | 反向污染训练样本 |

核心约束：

1. LLM/Agent 只能生成候选信号、证据摘要和解释，不直接拥有实盘执行权。
2. 风控和订单执行必须由确定性代码完成。
3. 每次决策必须记录数据版本、特征版本、模型版本、规则版本和证据时间。
4. 回测、模拟盘、影子模式和实盘共享同一套撮合、费用、滑点和仓位语义。
5. 所有收益展示默认使用扣除费用、滑点、资金费率和冲击成本后的净收益。

## 四、详细设计

### 4.1 回测可信账本设计

目标：让回测结果能用于策略比较、模型晋级和风控评估，而不是只作为展示结果。

| 对象 | 设计要求 |
|------|----------|
| `cash` | 表示可用现金，平仓后的已实现盈亏已经反映在 cash 中 |
| `realized_pnl` | 用作累计归因字段，不应在总权益计算中再次与 cash 相加 |
| `unrealized_pnl` | 只来自未平仓仓位的 mark-to-market |
| `fee` | 以订单成交额为基础计算，并在发生时扣减 |
| `slippage` | 每笔成交保留 bps 与金额，进入绩效报告 |
| `margin` | 当前为 1x notional 简化模型；杠杆保证金应作为后续独立模型 |

必备测试：开多/开空、同向加仓、部分平仓、完全平仓、反手开新方向、手续费与滑点、杠杆保证金、随机成交序列资金守恒。

回测引擎通过以下条件前，不允许作为“自动晋级实盘”的依据：撮合测试覆盖全部订单生命周期；绩效指标测试覆盖净收益、回撤、费用、滑点；任意成交序列不得产生负数量仓位、重复记账或权益不守恒。

### 4.2 绩效与归因设计

| 类别 | 指标 | 当前状态 |
|------|------|----------|
| 收益 | 总收益、年化收益 | 已有 |
| 风险 | 最大回撤、回撤曲线 | 已有 |
| 风险调整收益 | Sharpe、Sortino、Calmar | 部分已有 |
| 成本 | 手续费、滑点、资金费率、冲击成本 | 手续费已有，滑点金额缺失 |
| 尾部风险 | VaR、CVaR、最大单笔亏损 | 未完成 |
| 基准比较 | Benchmark return、Alpha、Beta | 未完成 |
| 稳定性 | 回撤持续期、月度胜率、滚动 Sharpe | 未完成 |
| 校准 | Brier Score、Log Loss、校准曲线 | 未完成 |

实现要求：`SimulatedFill` 或归因对象应保存 `slippage_cost`；`PerformanceReport` 应扩展 `alpha`、`beta`、`var_95`、`cvar_95`、`max_drawdown_duration`；API 默认返回净收益，毛收益仅作为调试字段；每个指标必须有可手算样本的单元测试。

### 4.3 数据与特征仓库设计

最新代码已经新增 `feature_definitions`、`feature_values`、`market_regimes`、`ohlcv_daily`，并提供 `/features/definitions`、`/features/values`、`/features/regimes/*` API。

下一步需要补齐：

| 能力 | 要求 |
|------|------|
| 数据血缘 | 每个特征值记录数据源、源时间范围、计算版本、参数 hash |
| 数据质量 | freshness、缺口率、异常值率、回填状态 |
| 长期回填 | 至少 2 年 OHLCV，优先 BTC/ETH/SOL 与高流动性 USDT 永续 |
| 高频数据 | 逐笔成交、主动买卖量、订单簿快照、深度不平衡 |
| 衍生品数据 | 资金费率、持仓量、多空比、清算、基差 |
| 跨市场数据 | 多交易所价差、现货-永续-交割合约价差 |
| 事件数据 | 新闻、宏观、监管、链上事件结构化 |

特征分类：动量、波动率、成交量、微观结构、资金与基差、市场状态。当前 MVP 已覆盖 RSI、MACD、SMA、EMA、ATR、Bollinger width、realized volatility、return、volume ratio、funding rate、market regime；后续补 ADX、Donchian、VWAP、OBV、MFI、CVD、spread、depth imbalance、basis、OI change。

### 4.4 概率信号与决策卡设计

预测目标不应是“价格是多少”，而应输出给定周期的条件分布：

```text
r(t,h) = ln(P(t+h) / P(t))
```

模型输出：`p_up`、`p_down`、`p_flat`、`q10/q50/q90`、`expected_volatility`、`mae_estimate`、`uncertainty`。

净期望：

```text
EV = p_up * mu_up + p_down * mu_down + p_flat * mu_flat - fee - slippage - funding - impact
```

交易必须同时满足：`EV > EV_min`、`P(EV > 0) > p_min`、`CVaR_95 < risk_budget`、`data_freshness == pass`。

决策卡字段：建议动作、预测周期、概率分布、收益区间、净期望、最坏情形、仓位建议、适用状态、支持证据、反对证据、样本表现、数据血缘。

### 4.5 组合与风控设计

禁止继续使用“最大仓位 × 主观置信度 × 波动率调整”作为最终仓位。目标仓位应由以下约束共同决定：单笔风险预算、Fractional Kelly、波动率目标、相关性约束、流动性约束、组合回撤约束。

| 门禁 | 触发后动作 |
|------|------------|
| 数据过期 | 禁止新开仓，只允许减仓和平仓 |
| Agent 高分歧 | 降低仓位或观望 |
| 市场状态不适用 | 信号降级 |
| 净期望不足 | 禁止交易 |
| CVaR 超预算 | 禁止交易 |
| 连续止损 | 暂停策略 |
| 当日亏损超限 | 触发紧急停止 |

### 4.6 执行与审计设计

执行层必须支持幂等订单 key、outbox 事件表、订单状态机、审计日志、影子模式、模拟盘门禁和断路器。

订单状态机：

```text
created -> risk_checked -> submitted -> partially_filled -> filled/cancelled/rejected
```

审计日志必须回答：谁、何时、基于什么证据、通过什么规则生成和执行订单。

验收标准：每笔交易都有完整证据链；用户可以看到“为什么做、何时证明错误、为何退出”；数据过期、模型失配、Agent 分歧时明确提示“暂不建议交易”。

## 九、下一步优先级

短期最应该做的不是继续增加 Agent，而是按以下顺序收口：

1. 补齐绩效报告的滑点成本、CVaR、Alpha/Beta 和回撤持续期。
2. 清洗并接入 `backend/tests/agents/*.rs`，建立测试门禁。
3. 给特征仓库补数据质量与血缘字段。
4. 建立概率决策卡的后端数据结构。
5. 在自动晋级或实盘授权流程中加入回测可信等级和影子模式门禁。

一句话判断：**项目方向正确，最新代码已经开始从“AI 辅助分析”走向“可验证量化基础”。当前设计已经补齐为可执行路线图，但实现上仍应优先把账算对、指标算准、概率校准、风险组合化和证据可追溯做深。**
