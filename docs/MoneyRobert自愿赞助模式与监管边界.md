# MoneyRobert 自愿赞助模式与监管边界

> 版本日期：2026-06-29  
> 结论：可以采用自愿赞助，但不能依靠“赞助”名称绕开自动交易、投资建议或虚拟资产服务监管。

## 一、为什么不能绕开监管

监管通常综合判断：

- 系统是否替客户决定买卖。
- 是否拥有持续或酌情交易权限。
- 是否向交易所发送订单。
- 是否以此作为持续业务。
- 是否获得任何直接或间接经济利益。
- 经济利益是否与交易、盈利、账户规模或使用服务存在联系。

如果只在盈利后弹出赞助、建议按盈利百分比支付、赞助后获得更好策略或系统反复催促，监管和法院可能把它视为变相报酬。美国 SEC 对 compensation 的解释就包括“任何经济利益”，不局限于名为顾问费的款项。

即使真正完全免费，只要系统替客户执行交易，毛里求斯、塞舌尔以及客户所在地仍可能基于投资管理、虚拟资产服务或酌情交易活动要求许可。收费方式只是判断因素之一，不是唯一因素。

## 二、可以采用的赞助设计

### 必须满足

- 产品全部功能不以赞助为条件。
- 不赞助不降低策略质量、执行速度、额度、客服或风险控制。
- 固定金额，不按盈利金额、账户余额、交易次数或节省损失计算。
- 不显示“建议赞助盈利的 5%/10%”。
- 不默认选择，不自动扣款，不从交易账户划扣。
- 用户可以永久关闭赞助提示。
- 同一价值事件只提示一次，不使用红点、倒计时或亏欠文案。
- 赞助不产生会员、优先信号、独家策略、治理权或收益承诺。
- 赞助进入公司账户或合规支付渠道，不进入作者私人钱包。
- 收据写明“voluntary creator support”，财务上如实入账纳税。

### 不应采用

- “盈利 ¥10,000，建议赞助 ¥1,000”。
- 支付按钮默认勾选 10%。
- 未赞助就暂停自动交易。
- 赞助用户获得更高杠杆或更早信号。
- 将历史未赞助金额累计成“待支付”。
- 把赞助写进交易管理协议的对价条款。
- 要求客户签署“盈利后应自愿赞助”的承诺。
- 使用私人 USDT 地址或不可退款的链上收款隐藏收入。

## 三、触发时机

最稳妥的方式是不读取盈利结果，统一放在“关于作者”页面。

如果产品仍希望在盈利后提示，应满足：

1. 只在完整平仓且扣除费用后出现。
2. 文案仅说明“如果产品帮到了你，可以自愿支持作者”。
3. 不展示盈利与赞助金额的换算关系。
4. 只提供固定金额，例如 ¥18、¥68、¥188。
5. 同时提供同等醒目的“无需赞助并永久关闭提示”。
6. 不记录“欠费”，只记录是否关闭提示。

盈利后触发仍然比普通“支持作者”入口具有更高的报酬关联风险，最终是否可接受需要由经营地和客户所在地律师确认。

## 四、推荐产品文案

标题：

> 如果 MoneyRobert 帮到了你，可以自愿支持作者

说明：

> 系统不收取订阅费或盈利分成。赞助不是使用条件，也不会改变交易、策略、提醒或任何产品权益。

按钮：

- 自愿赞助 ¥18
- 自愿赞助 ¥68
- 自愿赞助 ¥188
- 不需要，并关闭以后提示

禁止使用：

- 成功费
- 应付金额
- 盈利分成
- 回馈系统 10%
- 支付后继续交易

## 五、公司与资金处理

- 赞助收款主体应是海外实际经营公司。
- 使用公司银行、受监管支付服务商或经律师确认的企业加密支付渠道。
- 如果以虚拟货币收款，需要单独判断是否触发钱包、转账、Travel Rule 和 VASP 义务。
- 收入按当地会计和税法申报，不能因为名称是 donation 就默认免税。
- 应保存用户主动选择、提示版本、金额、退款和关闭提醒记录。
- 客户交易账户、交易所 API 和赞助支付完全隔离。

## 六、对出海成本的真实影响

赞助模式可以取消：

- 盈利服务费计算和应收账款。
- 自动划费和收费争议流程。
- 合作方之间的业绩报酬分账。
- 部分高水位线收费行政成本。

但仍不能取消：

- 交易系统许可分析。
- 客户所在地准入。
- KYC/AML 与制裁筛查。
- 适当性和风险揭示。
- 算法治理、交易风控和审计日志。
- 网络安全和数据保护。

因此，赞助模式能降低收费合规和运营成本，但不能把自动交易产品变成普通无监管软件。

## 七、最低风险替代产品

如果目标是在牌照前验证市场，最安全的 MVP 是：

- 系统只生成分析和模拟订单。
- 用户每笔交易自行确认。
- MoneyRobert 不保存交易 API Secret，不自动发送订单。
- 不托管、不转账、不归集资产。
- 赞助入口常驻“关于作者”，不以盈利触发。

这会显著降低“酌情管理账户”的风险，但是否仍构成投资建议要按客户所在地确认。
