       AVG(credibility_score) as avg_credibility,

### 2.4 数学验证：仓位 + 杠杆 + 单笔亏损

假设账户 10000 USDT，`max_position_size_percent=3`、`max_leverage=2`、`max_single_trade_loss_percent=1.5`：

```
position_value = 10000 × 3% = 300 USDT
notional = 300 × 2 = 600 USDT（实际敞口）
max_loss = 10000 × 1.5% = 150 USDT
required_stop_loss_pct = 150 / 600 = 25%（即价格反向变动 25% 才止损）
```

⚠️ 显然 25% 止损太宽。要么减小仓位、要么减小杠杆、要么收紧单笔亏损上限。

**正确配置示例**：

```
max_position_size_percent = 3
max_leverage = 2
max_single_trade_loss_percent = 1.0

position_value = 10000 × 3% = 300
notional = 600
max_loss = 10000 × 1% = 100
stop_loss_pct = 100 / 600 = 16.7%（仍然偏宽，但可接受）
```

## 第 3 章 概率校准（Brier Score）

### 3.1 为什么需要校准

AI 输出的"置信度 0.7"**不等于"真实概率 70%"**。可能 AI 说 0.7 时实际只有 55% 概率会涨（过度自信）。

校准就是用历史数据拟合一个映射：

```
原始置信度 p_raw → 真实概率 p_calibrated
```

线性校准公式（v1.8 实现）：

```
p_calibrated = slope × p_raw + intercept
```

### 3.2 Brier Score 是什么

Brier Score 衡量概率预测的准确度：

```
Brier = (1/N) × Σ (p_forecast - outcome)²
```

- `p_forecast`：预测概率
- `outcome`：实际结果（1 = 涨，0 = 跌）
- `N`：样本数

| Brier | 含义 |
|-------|------|
| 0.0 | 完美预测 |
| 0.33 | 接近随机（接近 33% 是因为三分类） |
| 0.5 | 比随机还差 |

**判定**：`brier_score < 0.33` 才认为校准数据可靠（[decision_engine.rs](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/signals/decision_engine.rs) 中 `is_reliable()` 方法）。

### 3.3 积累校准样本

校准需要 ≥ 30 个决策卡样本。来源：

1. paper 模式自动交易积累
2. 手动生成决策卡
3. 历史回测中产生的预测

查询当前样本数：

```sql
SELECT symbol, 
       COUNT(*) as sample_size,
       MIN(created_at) as earliest,
       MAX(created_at) as latest
FROM decision_cards
GROUP BY symbol;
```

### 3.4 生成校准报告

#### API 方式

```bash
curl http://localhost:8001/api/v1/signals/calibration/BTC-USDT-SWAP \
  -H "Authorization: Bearer $TOKEN"
```

返回 `CalibrationResponse`：

```json
{
  "symbol": "BTC-USDT-SWAP",
  "brier_score": 0.28,
  "calibration_slope": 0.85,
  "calibration_intercept": 0.05,
  "sample_size": 45,
  "is_reliable": true
}
```

#### 手动计算校准报告

```bash
curl -X POST http://localhost:8001/api/v1/signals/calibration/BTC-USDT-SWAP/compute \
  -H "Authorization: Bearer $TOKEN"
```

会重新基于历史决策卡计算 Brier Score 和校准曲线。

### 3.5 校准如何影响决策

`DecisionEngine.decide()`（[decision_engine.rs](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/signals/decision_engine.rs)）的执行流程：

```
1. 加载 signal.confidence（原始置信度）
2. 加载 calibration（CalibrationData）
3. if calibration.is_reliable():
     p_win = calibration.calibrate(signal.confidence)
     # 即 p_win = slope × confidence + intercept
   else:
     p_win = signal.confidence（使用原始）
4. 用 p_win 计算 EV、CVaR、决策
```

### 3.6 校准数据存储位置

校准数据存储在 `signal_calibration_reports` 表（[migrations/018:174-205](file:///d:/wan_workspase/MoneyRobert-Pro/backend/migrations/018_quantitative_foundations.sql)）：

```sql
SELECT symbol, brier_score, calibration_slope, calibration_intercept, 
       sample_size, computed_at
FROM signal_calibration_reports
WHERE symbol = 'BTC-USDT-SWAP'
ORDER BY computed_at DESC
LIMIT 5;
```

### 3.7 校准不达标的处理

如果 `brier_score ≥ 0.33` 或 `sample_size < 30`：

- 系统继续使用原始置信度（不校准）
- 决策质量取决于 LLM 本身
- 建议：继续积累样本，不要急着上 demo/live

## 第 4 章 EV/CVaR 模型详解（v1.8 增强版）

### 4.1 净期望 EV 公式

```
EV = p_up × mu_up + p_down × mu_down + p_flat × mu_flat
     - fee_cost - slippage_cost - funding_cost - impact_cost
```

其中：

- `mu_up / mu_down / mu_flat`：三场景下的预期收益（来自 Q10/Q50/Q90）
- `fee_cost`：手续费 = notional × fee_rate
- `slippage_cost`：滑点 = notional × slippage_bps / 10000
- `funding_cost`：资金费率累计（v1.8 新增）
- `impact_cost`：市场冲击（暂未实现）

### 4.2 资金费率成本计算（v1.8 新增）

```rust
let funding_rate_bps = input.funding_rate_bps
    .unwrap_or(self.config.default_funding_rate_bps);
let holding_sec = signal.expected_holding_period_sec
    .map(|v| v as f64)
    .unwrap_or(self.config.default_holding_period_sec);
let funding_periods = (holding_sec / 28800.0).max(1.0);
let funding_cost_pct = (funding_rate_bps × funding_periods) / 10000.0;
```

**关键参数**：

- `funding_rate_bps`：资金费率（bp，1 bp = 0.01%）
- `holding_sec`：预期持仓时间（秒）
- `28800`：资金费率结算周期（8 小时 = 28800 秒）
- `funding_periods`：持仓期间结算次数（最少 1 次）

**示例**：

```
funding_rate_bps = 10 bp (0.1%)
holding_sec = 14400 (4 小时)
funding_periods = 14400 / 28800 = 0.5 → max(1.0) = 1
funding_cost_pct = (10 × 1) / 10000 = 0.001 = 0.1%
```

⚠️ 持仓 < 8 小时也会算 1 次资金费率（保守估算）。

### 4.3 CVaR 改进公式（v1.8 增强）

旧版（简化）：

```
CVaR = 1.65 × asset_vol × 0.05
```

新版（v1.8）：

```rust
let sigma_daily = asset_vol / (365.0_f64).sqrt();
let holding_days = (holding_sec / 86400.0).max(1.0).min(30.0);
let z_cvar = 2.063;  // 95% CVaR 正态分布分位数
let estimated_position_pct = 0.05;  // 假设 5% 仓位
let cvar = z_cvar × sigma_daily × holding_days.sqrt() × estimated_position_pct;
```

**关键改进**：

1. `2.063`：正态分布 95% CVaR 的分位数（比 VaR 的 1.65 更严格）
2. `sqrt(holding_days)`：考虑持仓时间的 sqrt(t) 效应
3. `× estimated_position_pct`：CVaR 是相对于仓位的，不是绝对的

### 4.4 校准后概率如何使用

`DecisionOutput.confidence` 字段（[decision_engine.rs](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/signals/decision_engine.rs)）改用校准后的概率：

```rust
// 旧版：直接用原始 confidence
// output.confidence = signal.confidence

// 新版（v1.8）：用校准后的概率
output.confidence = p_win;
```

意味着如果 LLM 报 0.75 置信度，但校准后真实概率只有 0.62，系统会用 0.62 做决策（更保守）。

### 4.5 决策门禁的实际应用

`DecisionOrchestrator.orchestrate()` 调用 `DecisionEngine.decide()` 后，会按以下门禁判定：

```
if EV < ev_min:           拒绝（blocker: "EV 不足"）
if CVaR > risk_budget:    拒绝（blocker: "CVaR 超预算"）
if p_win < p_min:         拒绝（blocker: "概率不足"）
if data_freshness > 300:  拒绝（blocker: "数据过期"）
```

任何一项触发，决策卡 `blockers` 字段会记录原因，`trust_level` 可能降级。

## 第 5 章 组合风控 PortfolioRiskEngine

### 5.1 组合风控源码位置

### 5.2 四项组合风控检查

#### 检查 1：CVaR 预算

```rust
// 组合 95% CVaR ≤ 总风险预算
if portfolio_cvar > total_risk_budget {
    // 按比例缩减所有仓位
    let scale = total_risk_budget / portfolio_cvar;
    positions.iter_mut().for_each(|p| p.size *= scale);
}
```

#### 检查 2：风险贡献集中度

```rust
// 单一资产风险贡献 ≤ 40%
for asset in assets {
    if asset.risk_contribution > 0.4 * portfolio_var {
        // 缩减该资产仓位
    }
}
```

#### 检查 3：相关性约束

```rust
// 高相关性资产（> 0.7）总仓位 ≤ 上限
let correlated_group = assets.iter()
    .filter(|a| a.correlation_with(other) > 0.7)
    .collect();
if correlated_group.total_position > correlated_limit {
    // 缩减
}
```

#### 检查 4：流动性约束

```rust
// 仓位 ≤ 24h 成交量的 1%
for asset in assets {
    if asset.position_value > 0.01 * asset.volume_24h {
        // 缩减
    }
}
```

### 6.3 资格检查逻辑

```rust
pub fn check_promotion_eligibility(stats: &RollingStats, current_level: i32) -> PromotionEligibility {
    let next_level = current_level + 1;
    let requirements = get_level_requirements(next_level);
    
    let mut missing = vec![];
    if stats.total_trades < requirements.min_trades { missing.push("trades"); }
    if stats.win_rate < requirements.min_win_rate { missing.push("win_rate"); }
    if stats.profit_loss_ratio < requirements.min_profit_loss_ratio { missing.push("pl_ratio"); }
    if stats.running_days < requirements.min_running_days { missing.push("running_days"); }
    if stats.max_drawdown_percent > requirements.max_drawdown_percent { missing.push("drawdown"); }
    if stats.consecutive_stop_losses > requirements.max_consecutive_losses { missing.push("consecutive_losses"); }
    
    PromotionEligibility {
        eligible: missing.is_empty(),
        missing_requirements: missing,
        ...
    }
}
```

## 第 8 章 降级触发与恢复

### 8.1 降级触发条件

源码：[promotion.rs:389-434](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/agents/promotion.rs)，`check_demotion` 函数。

满足**任一**条件即降级（Level → Level-1）：

| 条件 | 含义 |
|------|------|
| `max_drawdown_percent > requirements.max_drawdown_percent × 1.5` | 最大回撤超限 1.5 倍 |
| `consecutive_stop_losses > requirements.max_consecutive_losses × 2` | 连续止损超限 2 倍 |
| `current_balance / initial_balance < 0.5` | 账户资金腰斩 |

### 8.2 降级示例

当前 Level 2（autonomous），要求 `max_drawdown_percent ≤ 8.0`：

```
实际 drawdown = 13%（> 8.0 × 1.5 = 12.0）
→ 触发降级
→ Level 2 → Level 1（demo）
→ consecutive_stop_losses 重置为 0
→ 记录 demotion_records
```

### 8.3 降级记录查询

```sql
SELECT id, config_id, from_level, to_level, 
       reason, triggered_at
FROM demotion_records
WHERE config_id = '你的_config_id'
ORDER BY triggered_at DESC;
```

### 8.4 降级后恢复

降级后：

1. mode 自动改为降级后的模式
2. `consecutive_stop_losses` 重置为 0
3. 需要重新满足当前等级的晋级条件才能再次晋级
4. 晋级流程仍需 14 天观察 + 人工审批

### 8.5 紧急停止 vs 降级

| 操作 | 触发方式 | 结果 |
|------|---------|------|
| 紧急停止 | 人工 API 调用 | status=stopped，循环退出，**等级不变** |
| 降级 | 系统自动检查 | 等级 -1，mode 改变，**循环继续** |

紧急停止是"暂停"，降级是"等级下调"。

## 第 9 章 决策调优台

### 9.1 决策调优台位置

设置页 → 决策调优台（`http://localhost:3000/settings`）

### 9.2 核心配置项

#### 部门权重

| 配置 | 默认 | 调优建议 |
|------|------|---------|
| Technical | 0.35 | 趋势行情 ↑ 0.40 |
| Capital | 0.35 | 资金费率极端 ↑ 0.40 |
| News | 0.30 | 宏观事件期 ↑ 0.35 |

权重和不必为 1，系统会自动归一化。

#### 最低方向优势（floor / ceiling）

- `floor`：多空观点差异低于此值 → 观望（建议 0.1）
- `ceiling`：多空观点差异高于此值 → 强信号（建议 0.3）

#### 最低数据质量

- 低于此等级不交易（建议 `good`）
- 可选：`excellent` / `good` / `fair` / `poor`

#### 历史可靠度强度

- 决定记忆系统（episodic_memory）的权重
- 0 = 不参考历史，1 = 完全按历史
- 建议 0.3-0.5

#### AI 冲突策略

当多 Agent 观点严重冲突时：

- `score_wins`：按置信度加权决定（默认）
- `hold_on_conflict`：冲突时观望（保守）

### 9.3 调优案例

#### 案例 1：趋势行情中提升技术部权重

```
Technical = 0.40
Capital = 0.30
News = 0.30
最低方向优势 floor = 0.05（趋势更易一致）
AI 冲突策略 = score_wins
```

#### 案例 2：宏观事件期保守化

```
Technical = 0.25
Capital = 0.30
News = 0.45
最低方向优势 floor = 0.15（要求更高一致性）
AI 冲突策略 = hold_on_conflict
最低数据质量 = excellent
```
