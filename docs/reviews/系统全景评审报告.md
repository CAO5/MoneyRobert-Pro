## 0. 执行摘要(Executive Summary)

本报告从三个互补的专业视角,对 MoneyRobert Pro 进行交叉评审。三视角独立得出的结论高度一致,指向同一组**阻塞性问题**:

| 视角 | 核心关切 | 一句话结论 |
|------|----------|------------|
| **基金经理** | 能不能用、会不会亏钱 | 实盘下单完全绕过风控,风控只活在回测里。**不能碰真钱。** |
| **金融学家** | 模型理论成不成立 | 量化内核建立在 3 个被主流文献证伪的假设上(正态分布/线性校准/常数相关)。**风险度量在尾部会失效。** |
| **产品经理** | 给谁用、能不能完成任务 | 35 个页面、定位不清、信任缺失。**像目录而非产品。** |

### 0.1 三视角共识的最高优先级结论(红线项)

以下 3 项被**两个或三个视角同时**判定为阻塞性问题,优先级最高,必须在任何新功能开发前解决:

1. **实盘交易绕过风控闭环**(基金经理 P0 / 产品经理 P0)
   — `trading.rs::create_order` 直接调 OKX,不经 RiskEngine/Orchestrator。
2. **核心风险/收益模型的理论假设不成立**(金融学家核心 / 基金经理 严重)
   — 正态 CVaR、伪 EV 公式、硬编码概率权重。
3. **产品定位分裂、核心旅程不闭环**(产品经理核心 / 基金经理 严重)
   — 无 Persona、无新手引导、决策卡反人类(手输概率)。

> **决策建议**:在补齐红线项之前,**禁止任何形式的实盘资金接入**;任何拉新/增长动作都应暂停。

## 1. 评审方法与范围

### 1.1 评审方法

采用**多视角交叉评审(Triangulated Review)**,避免单一专业视角的盲区:

- **基金经理视角**:以"能否签字放行上真金白银"为验收标准,关注执行链路、资金安全、风控闭环。
- **金融学家视角**:以"理论与统计是否成立"为验收标准,关注概率假设、风险度量、推断统计。
- **产品经理视角**:以"目标用户能否完成核心任务、是否有商业价值"为验收标准,关注定位、旅程、信息架构。

每个问题给出**代码级证据**(文件:行号),确保结论可追溯、可验证。

### 1.2 评审范围

- **后端**:`backend/src/`(routes/agents/backtest/signals/features/orchestration/exchanges)

- **数据库**:`backend/migrations/`(0xx 系列)

- **超出范围**:CI/CD、生产部署运维、第三方依赖安全审计(建议另行专项)

### 1.3 评审局限

- 未执行运行时测试(未跑回测、未连交易所沙箱),结论基于**静态代码审查 + 现有文档交叉验证**;

## 2. 视角一:基金经理评审

> **验收标准**:如果这是我的团队要拿来管真金的系统,我能不能签字放行?

### 2.1 一句话结论

> 这套系统目前是一个**"看起来像量化基金,但执行链路是断的"**的 AI 交易工作台。前端页面和 Agent 数量给人"很完整"的错觉,但从基金经理的角度看,**它的钱袋子和风控闸门是分开的、各自为政的**。在没有补齐执行-风控闭环之前,绝不能上真金白银。

### 2.2 致命级问题(直接导致亏损 / 爆仓)

#### 2.2.1 实盘下单完全绕过风控 —— 一票否决

`backend/src/routes/trading.rs::create_order`:
- 直接拿用户请求里的 `quantity / leverage / stop_loss` 透传给 OKX;
- **没有调用 `DecisionOrchestrator`、`RiskEngine`、`PortfolioRiskEngine` 中的任何一个**;
- 没有 CVaR 预算检查、没有组合相关性检查、没有日亏熔断、没有流动性约束。

系统里那套花大力气写的"概率决策卡 + Fractional Kelly + 组合 CVaR"(`position_sizing.rs`、`portfolio_risk.rs`),**在真实下单的那一刻全部失效**。风控不在执行链路里,等于没有风控。

#### 2.2.2 "自动交易"是假的 —— 根本没真的下单

`backend/src/agents/autonomous.rs::execute_trade`:
```rust
let success = true; // Trade was executed successfully
let pnl_percent: Option<f64> = None;
```
- 只写日志、更新内部计数,**没有调 OKX、没有调 paper executor、没有产生持仓**;
- `success = true` 被喂给 `risk_watchdog.record_trade_outcome`,**风控看门狗收到的"胜率"是 100% 写死的**——任何基于"历史胜率"的自适应逻辑(包括 Kelly)全部失真。

`perform_risk_checks`(`autonomous.rs:1056`)只检查 `confidence >= 阈值` 和 `symbol 在白名单`,**这不是风控,是过滤条件。**

#### 2.2.3 风控引擎本身太薄,挡不住真实风险

`backend/src/backtest/risk_engine.rs` 的 `RiskEngine` 只有 3 个检查:
- 日内亏损(`max_daily_loss_pct=3%`)
- 单资产名义敞口上限(`max_single_position_pct=10%`)
- 总杠杆上限(`max_total_leverage=3x`)

**基金经理会立刻问、但系统答不上来的问题**:
- 组合层面的 CVaR / VaR 在哪?(只在回测 `portfolio_risk.rs`,**实盘没有**)
- 单笔最大亏损在哪?连续止损熔断?数据过期只允许减仓?相关性约束?
- 文档自列 7 个风控门禁(数据过期/高分歧/状态不适用/EV 不足/CVaR 超限/连续止损/当日亏损),**实盘路径一个都没接**。

### 2.3 严重级问题(模型可信度 / 资金管理逻辑错误)

#### 2.3.1 仓位模型把"预期收益"当成"历史平均亏损" —— Kelly 公式用错

`decision_orchestrator.rs:165-168`:
```rust
let avg_win = expected_return_bps.max(0.0) / 10000.0;
let avg_loss = expected_return_bps.abs().min(500.0) / 10000.0;
```
喂给 Kelly `f* = (p·b − q)/b`。Kelly 公式里的 `avg_win/avg_loss` 应是**历史交易统计出的盈亏比**,这里却用了**当前信号的预期收益率**(对未来收益的点估计)。`PositionSizingEngine::estimate_from_history`(`position_sizing.rs:207`)明明写好了正确的历史统计方法,**编排器根本没调用它**。结果算出的"Kelly 仓位"在数学上没有意义。

#### 2.3.2 杠杆计算逻辑混乱

`position_sizing.rs:167`:
```rust
let leverage = (position_pct / max_position_pct * max_leverage)
    .min(max_leverage).max(1.0)
```
杠杆应由保证金占用、波动率、CVaR 预算决定,**而不是"仓位占 max 的百分比"线性映射**;`.max(1.0)` 把所有小仓位杠杆抬到 1x 起步,语义不明。

#### 2.3.3 Agent 置信度是主观打分,不是校准概率

- `agents.rs` 每个 Agent 的 `credibility_score` 是**硬编码常量**;
- `department_weights`(技术 0.35 / 资金 0.35 / 新闻 0.30)写死;
- `decision_orchestrator.rs:138` `calibration: None`(TODO 未接入),**校准数据根本没进决策**。

#### 2.3.4 funding_rate / 滑点 / 冲击成本在实盘没入账

- `decision_orchestrator.rs:136` `funding_rate_bps: None`(TODO);
- 实盘市价单不估算滑点和冲击成本;
- 回测里有滑点入账,实盘没有 → **回测收益和实盘收益系统性不可比**。

#### 2.3.5 两套配置参数互相打架

| 来源 | max_leverage |
|------|--------------|
| `FundManagerConfig` | 5x |
| `RiskConfig` | 3x |
| `auto_trading` 默认 | 1x |

### 2.4 结构性问题(回测可信度 / 可复现性)

#### 2.4.1 三套语义不统一:回测 ≠ 模拟盘 ≠ 实盘

- **回测**:接了 Orchestrator + LedgerService + PortfolioRiskEngine;
- **模拟盘**(`paper_trading.rs`):走 `PaperTradingExecutor`,**未接 orchestrator、未接组合风控**;
- **实盘**(`trading.rs`):直接透传 OKX,**啥风控都没有**。

> 回测里 Sharpe 2.0、回撤 5% 的策略,**在模拟盘和实盘里根本无法复现**。

#### 2.4.2 没有样本外验证 / Walk-forward 接入主链路

`backtest/walk_forward.rs` 存在但未被自动交易/决策编排调用;没有按 regime 分组的 OOS 绩效;没有过拟合检测。

#### 2.4.3 缺长期历史数据 + 特征版本血缘未闭环

文档承认特征仓库刚启动,"长期回填"是 TODO。

### 2.5 流程 / 治理问题

- **没有归因 → 仓位调整的闭环**:绩效能统计,但没人定期更新 Kelly 输入;
- **没有"谁能授权上实盘"的强制门禁**:晋级流程(回测可信等级→影子→模拟→实盘)在代码层无强制。

### 2.6 基金经理视角的行动清单

| 优先级 | 动作 | 阻断风险 |
|--------|------|----------|
| P0-1 | `trading.rs::create_order` 强制接入 `DecisionOrchestrator.orchestrate()`,`can_execute()==false` 拒单 | 裸奔下单 |
| P0-2 | `autonomous.rs::execute_trade` 真正调用 OKX/paper executor,删 `success=true` | 假交易污染统计 |
| P0-3 | 实盘下单经 `RiskEngine` + `PortfolioRiskEngine`,写 `risk_decisions` 审计 | 组合级风控落地 |
| P0-4 | 修正 orchestrator Kelly 输入:用 `estimate_from_history`,禁用 expected_return 当 avg_loss | 仓位数学正确 |
| P1-1 | 模拟盘/实盘统一走 `LedgerService`,与回测共享撮合/费用/滑点语义 | 三层可对账 |
| P1-2 | funding/滑点/冲击成本接入 EV,决策卡默认展示净期望 | 成本不被低估 |
| P1-3 | 校准数据从 DB 自动读入 `DecisionInput.calibration` | 概率可信 |
| P1-4 | Walk-forward 接入主链路,按 regime 分组展示 OOS | 防过拟合 |

### 2.7 最尖锐的一句

> 文档自评:"当前系统更准确的定位是:**AI 辅助交易工作台 + 确定性风控执行系统**"。
> 问题在于:**第二半还没实现**。风控引擎写好了,但只在回测里跑;实盘直接绕过。对基金经理而言,**风控不在下单路径里 = 系统没有风控**。

## 3. 视角二:金融学家评审

> **验收标准**:模型的理论假设成立吗?统计推断有效吗?风险度量在数学上自洽吗?

### 3.1 一句话结论

> 从金融学角度看,这套系统的"量化内核"建立在三个被主流文献证伪的假设上:**收益正态分布、线性可校准概率、常数相关性**。它的风险模型在最关键的尾部和危机场景下会失效;它的收益模型(EV)在数学上不是期望价值;它的回测结论没有任何统计推断支撑。

### 3.2 核心理论错误(违反金融学基本原理)

#### 3.2.1 正态分布假设用在最不该用的地方 —— CVaR

`decision_engine.rs:254-257`、`portfolio_risk.rs:138-147`:
```rust
let z_cvar = 2.063; // 95% CVaR 正态分布分位数
let cvar = z_cvar * sigma_daily * holding_days.sqrt() * position_pct;
```

**这是金融学教科书第一课就否决的做法。** 加密资产收益是**重尾分布(fat-tailed)**,左偏、厚尾显著,5σ 事件的发生频率是正态假设下的几十到上百倍。用正态 CVaR 会**系统性低估真实尾部风险一个量级**:系统标榜的"5% CVaR 预算"在真实尾部下可能实际是 15%-20% 损失。

系统内部**理论不自洽**:`performance_engine.rs:155` 的 `var_cvar_95` 用经验分位数(历史模拟法),风控却用正态——**同一风险的估计差一个数量级**。

**正确做法**:历史模拟法 / Student-t 拟合 / EVT(极值理论)。

#### 3.2.2 EV(期望价值)公式在数学上不成立

`decision_engine.rs:207-241`:
```rust
let p_loss = (1.0 - p_win) / 2.0;   // 任意拆分
let mu_win = expected_return_bps.max(0.0) / 10000.0;
let mu_loss = expected_return_bps.abs().min(500.0) / 10000.0; // 盈亏幅度都来自同一点估计
```

金融学上 EV 必须基于收益的**完整条件分布** `E[r|信号] = ∫ r·f(r|信号)dr`。问题:
1. **概率拆分任意**:`p_loss = (1−p_win)/2` 的 50/50 划分无理论依据;
2. **用点估计代替分布**:假设"盈利幅度 = 亏损幅度 = 预期收益",经济上荒谬;
3. **三分类强行退化**:文档要求输出 `p_up/p_down/p_flat + q10/q50/q90`,实现里只有 `p_win` 一个真实概率。

**结论:这个 EV 在数学上不是期望价值,是单点输入伪装的分布推理。** 建立在它之上的 Kelly/仓位/EV 门禁都继承其无效性。

#### 3.2.3 校准函数用线性插值,违反概率公理 + data snooping

`calibration.rs:60` 线性校准 + `is_reliable()` 仅要求 `brier < 0.33`:
- 线性校准不保证单调、不保证 [0,1],clamp 引入平台段(预测无区分度);
- `train_from_predictions`(`calibration.rs:910`)**无 train/calibration/test 三分**,用同一批数据拟合又报告质量 → **系统性高估校准效果(data snooping)**;
- Brier < 0.33 作为"可靠"门槛**方向可能搞反**:二分类随机预测期望 Brier ≈ 0.25,0.33 比 random 还差却被判可靠;
- 实现了 Platt/Isotonic,但**决策链路只接了最弱的 Linear**。

### 3.3 风险与收益度量的统计有效性

#### 3.3.1 Sharpe 假设无风险利率为 0,且未处理自相关

`performance_engine.rs:55-68`:
```rust
(mean / std) * (365.0_f64).sqrt()  // 无风险利率 = 0
```
- 忽略 USDT 资金成本,**高估 Sharpe**;
- 未做 Lo(2002)自相关修正:`SR_adj = SR/√(1+2Σρ_k/T)`,**趋势策略正相关时标准 Sharpe 高估真实风险调整收益**。

#### 3.3.2 年化口径混用,回撤基于含噪声的盯市权益

- 波动率年化用 `√365`、收益用 365 复利,日历日/交易日口径未严格区分;
- `max_drawdown` 基于 `peak`,但权益含未实现盈亏的 mark-to-market 噪声,**瞬时插针会制造虚假回撤峰**,污染 MaxDD/Calmar/回撤持续期。

### 3.4 市场状态(Regime)模型的理论缺陷

#### 3.4.1 Regime 是规则阈值,不是真正的状态切换模型

`regime.rs:314-355` 是 `if-else` 阈值规则。经典 regime 核心是**状态持续性 + 转移概率**(HMM / MS-AR)。这里:
- 置信度拍脑袋(`(adx/50)*0.7+0.3`,Crisis 固定 0.9);
- 无转移矩阵、无状态持续性建模;
- 决策引擎对 regime 的依赖(降 EV 阈值、乘仓位系数)**远超该分类器的可靠性**。

#### 3.4.2 ADX 计算错误(返回的是 DX,不是 ADX)

`regime.rs:270-271` 注释自承:
```rust
// ADX 是 DX 的平滑平均，简化为最后 DX 值
dx.min(100.0)
```
返回 DX 而非 ADX,趋势强度估计**远比真实 ADX 波动剧烈**,"ADX>25=有趋势"在噪声中频繁翻转。

#### 3.4.3 波动率分位数用窗口内自己的数据排名

`regime.rs:167-179` 当前波动率与**同一窗口内**的滚动波动率排名。若整个窗口都是高波动期,当前值分位数不高 → 误判"非高波动"。需**长期历史基准**。

### 3.5 组合理论的根本缺失

#### 3.5.1 常数相关系数,危机时失效

`portfolio_risk.rs:109-129` 用常数相关矩阵。金融学铁律:**相关性在压力下趋向 1**(2008/2020/2022 加密寒冬均验证)。常数相关 + 正态 CVaR,**在最需要时系统性低估组合风险**。理论应至少:条件相关(DCC-GARCH)/ 分 regime 相关 / copula 尾部相关。

#### 3.5.2 风险贡献均衡公式对,但输入无效

`portfolio_risk.rs:154-191` 边际风险贡献 MRC 数学正确,但依赖的 `Σ` 来自常数相关 + 常数波动率,无 GARCH 时变特征。Risk Parity 流派强调输入误差被 `w'Σw` 放大,应做 robust optimization。

### 3.6 计量经济学 / 假设检验层面

#### 3.6.1 没有任何策略显著性的统计检验

- 无 t-检验/p-value(Sharpe 是否显著异于 0);
- 无 White's Reality Check / Hansen SPA(多策略防 data snooping);
- 无 Deflated Sharpe Ratio(Bailey & López de Prado 2014)。

`trust_engine.rs` 的可信等级门禁只检查**描述性**(样本量、时间跨度),不检查**推断性**(显著性)。Sharpe 2.0 但样本 30 笔,95% CI 可能横跨 [-0.5, 4.5],**和没赚钱无法区分**。

#### 3.6.2 数据窥探(data snooping)无防护

- `walk_forward.rs` 未接主链路;
- 13 Agent × 多 regime × 多参数组合隐式产生大量回测;
- 无样本外/OOS 报告、无多重检验校正。

### 3.7 金融学家视角的裁决矩阵

| 维度 | 裁决 |
|------|------|
| 风险度量(CVaR) | 正态分布在加密可证伪,真实尾部被低估一个量级。**理论硬伤,非调参可解。** |
| 收益预测(EV) | 非真正期望价值,单点伪装的分布推理。**其上所有决策理论根基无效。** |
| 校准 | 实现了正确方法但只接最弱线性,且无严格三分,存在 data snooping。 |
| Regime | 规则阈值冒充状态模型,无转移概率,ADX 算成了 DX。 |
| 组合 | Markowitz 1952 + 常数相关,无 tail dependence、无稳健优化,危机时失效。 |
| 统计推断 | 零。无 p-value、无多重检验校正、无 Deflated Sharpe。回测"显著"是叙事。 |

## 4. 视角三:产品经理评审

> **验收标准**:这是给谁用的?用户能不能完成核心任务?哪些是真需求、哪些是伪需求?

### 4.1 一句话结论

> 从产品经理角度看,MoneyRobert Pro 当前的核心问题**不是功能不够,而是功能太散、定位不清、信任缺失**。它像一本"量化交易百科全书的目录"——什么都有目录,但每一章都只写了个开头。

### 4.2 产品定位诊断:最根本的问题

#### 核心矛盾:想"既给机构又给散户",两边不讨好

| 表现 | 散户视角 | 机构/专业交易员视角 |
|------|----------|---------------------|
| 35 个导航项 | 信息过载,看不懂从哪入手 | 颗粒度太粗,要的还是 TradingView |
| 概率决策卡、Kelly、反事实 | 看不懂、不敢用 | 太"教科书",缺真实因子库 |
| AI 辩论 13 Agent | 像看戏,不知如何决策 | 怀疑可靠性,看不到统计依据 |
| 实盘直连 OKX | 想要"一键跟单" | 想要托管/多账户/API key 管理 |

**PM 第一判断:产品没有明确 Persona。** 功能堆砌而非产品,没有"非用你不可"的闭环。这解释了前三轮那么多问题能并存——**因为没有核心用户旅程来验收,每个功能都是孤岛。**

### 4.3 按用户旅程诊断:三大主流程的真实状态

#### 旅程 A:新用户从注册到第一笔交易 —— 不闭环 ❌

| 步骤 | 状态 | 问题 |
|------|------|------|
| 注册/登录 | ✅ 有 | — |
| 新手引导 | ❌ 无 | 35 页直接拍脸上,无 onboarding |
| 风险测评/适当性 | ❌ 无 | 合规硬伤 |
| 接入交易所/API key | ⚠️ 半 | OKX key 填在哪?设置页 11 处 TODO |
| 模拟盘熟悉 | ⚠️ 半 | PaperTrading 调 `/agent/simulation`,语义混乱 |
| 第一笔真实交易 | ❌ 危险 | 实盘绕过风控,可能直接亏钱 |

**新用户漏斗在第 2 步就漏光。** 没有 onboarding → 乱点 → 撞进实盘 → 亏钱 → 流失。这是负向 PMF。

#### 旅程 B:交易员日常交易 —— 价值点错位 ⚠️

核心任务:看行情 → 形成判断 → 下单 → 管理持仓 → 复盘。当前系统把旅程打碎成 5 个独立页面,且:
- TradingPage(750 行)是核心,但绕过风控下单;
- **DecisionCard 是手动填概率生成**——PM 角度反人类:用户要 AI 决策,结果让他自己手输 p_up/p_down/q10/q90?决策卡是"给研究员演示的",不是"给交易员用的";
- 缺真实交易员高频功能:多窗口看盘、画线工具、自定义指标、条件单/网格单、止盈止损联动、移动止损。

**核心交易旅程产品力远不如 TradingView/OKX 原生。用户没有理由在你这下单。**

#### 旅程 C:研究员做策略研究 —— 最完整但最危险 ✅→❌

最完整的旅程(回测+归因+校准+反事实+walk-forward),也是 PM 视角最该砍掉的:
- 研究旅程用户(量化研究员)**不是核心付费用户**;
- 占 35 页中的 **12 个**(回测系+信号系);
- 更糟:回测/校准/Regime **理论是错的**,产出结论不可信。

**典型的"工程师自嗨":团队擅长什么就做什么,而非用户需要什么。应做减法。**

### 4.4 六大产品级问题

1. **35 个页面 = 无信息架构(IA)**:导航 7 组 35 项,无优先级层次。用户进来不知点哪。对比 Robinhood/TradingView 的聚焦。
2. **合规与风控的"双重真空"**:
   - 用户侧:无 KYC、无风险测评、无适当性;AutoTrading `mode` 可直接选 'live' 无拦截;多用户无资金隔离/操作审计。
   - 系统侧:实盘绕过风控,自动交易是假桩。**用户以为有风控实则没有——虚假安全感比没风控更糟。**
3. **核心价值主张(AI 决策)不可证伪、不可解释给终端用户**:Agent 输出无历史命中率标签(credibility 硬编码);决策卡概率手输;用户无法判断"这个建议靠不靠谱"。**信任是交易产品命脉,当前 AI 呈现增加而非降低决策负担。** 前端展示的 p_up/CVaR 后端是正态假设,却无"基于 XX 假设,误差 YY"的诚实标注。
4. **功能平铺,无 Aha Moment**:12403 行前端、35 页,首次进 Dashboard 一堆图表,无引导钩子。
5. **移动端 = 0**:全 PC 布局,加密核心场景(移动盯盘/移动止损)缺失。
6. **无运营/增长闭环**:无价格告警/持仓止损/AI 信号推送策略;无分享/社区/跟单;商业模式不明。

### 4.5 产品经理视角的战略级建议

1. **先定位,再补功能。** 必须先答:**专业个人交易员、散户、机构三选一。** 不聚焦的产品没有 PMF。
2. **信任优先于功能。** 交易产品护城河是信任。最大危机不是功能少,而是"看似有风控实则没有""看似是概率实则是正态点估计"。补新功能前先让"每个数字都可被信任、可被解释"。
3. **用"最小可信闭环"代替"最大功能集"。** 砍一半页面,集中资源做:**一个诚实标注假设的决策卡 + 一个真正接风控的模拟盘 + 一个移动端盯盘止损**。这三件做成,比 35 个半成品页面值钱十倍。

| 语义 | `success`=交易盈亏(simulation.rs/ai_analysis.rs 用);`was_correct`=决策方向对错(memory.rs/evolution.rs 用)。两者共存,语义不同 |
