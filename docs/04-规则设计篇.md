## 第 1 章 系统架构总览

### 1.1 六层架构

系统采用模块化单体设计，分为六层：

```
Market Data  →  Feature Store  →  Model/Signal  →  Portfolio/Risk  →  Execution  →  Attribution
   采集         特征仓库           信号与决策         组合风控           执行          归因
```

| 层级 | 职责 | 不应承担的职责 | 主要目录 |
|------|------|---------------|---------|
| Market Data | 采集、清洗、去重、时间对齐、质量标记 | 直接生成交易建议 | `src/collector.rs` |
| Feature Store | 计算、存储、版本化特征 | 临时拼接不可复现特征 | `src/features/` |
| Model/Signal | 输出概率、分位数、状态、解释 | 直接决定仓位或下单 | `src/signals/`、`src/agents/` |
| Portfolio/Risk | 组合约束、仓位预算、风险门禁 | 修改模型概率 | `src/backtest/risk_engine.rs`、`portfolio_risk.rs` |
| Execution | 幂等下单、撮合、订单状态机、审计 | 绕过风控执行 | `src/orchestration/ledger_service.rs`、`src/exchanges/` |
| Attribution | 绩效、归因、校准、复盘 | 反向污染训练样本 | `src/backtest/performance_engine.rs`、`src/signals/calibration.rs` |

### 1.3 数据流（一次完整交易）

```
1. collector.rs 采集 OKX 行情
       ↓
2. features/regime.rs 识别市场状态
       ↓
3. agents/debate.rs 启动多 Agent 辩论
       ↓  输出 FundManagerDecision
4. orchestration/decision_orchestrator.rs 编排
       ↓  调用 signals/decision_engine.rs（EV/CVaR）
       ↓  调用 backtest/risk_engine.rs（单笔风控）
       ↓  输出 can_execute() + final_position_pct
5. backtest/runner.rs 调用 apply_portfolio_risk_check()
       ↓  调用 portfolio_risk.rs（组合风控）
       ↓  按比例缩减仓位
6. orchestration/ledger_service.rs apply_fill()
       ↓  调用 matching_engine.rs 撮合
       ↓  调用 exchanges/okx.rs 下单（demo/live）
       ↓  记录 LedgerEntry
7. signals/store.rs 写入决策卡快照
       ↓
8. backtest/performance_engine.rs 更新绩效统计
```

### 1.4 五大核心约束（设计原则）

修改代码时必须遵守：

1. **LLM/Agent 只能生成候选信号、证据摘要和解释**，不直接拥有实盘执行权
2. **风控和订单执行必须由确定性代码完成**，不被 LLM 覆盖
3. **每次决策必须记录数据版本、特征版本、模型版本、规则版本和证据时间**
4. **回测、模拟盘、影子模式和实盘共享同一套撮合、费用、滑点和仓位语义**
5. **所有收益展示默认使用扣除费用、滑点、资金费率和冲击成本后的净收益**

#### 添加自定义风控规则

例如：添加"市场状态 = crisis 时禁止开仓"：

```rust
// 在 risk_engine.rs 中添加方法
pub fn check_market_regime(&self, regime: &MarketRegime) -> RiskCheckResult {
    if regime.regime == "crisis" {
        return RiskCheckResult::reject("market in crisis, no new positions");
    }
    if regime.confidence < 0.4 {
        return RiskCheckResult::reject("regime confidence too low");
    }
    RiskCheckResult::pass()
}

// 在 RiskEngine::check() 中调用
pub fn check(&self, signal: &Signal, account: &AccountState, regime: Option<&MarketRegime>) -> RiskCheckResult {
    // ... 原有检查 ...
    
    // 新增：市场状态检查
    if let Some(r) = regime {
        if let Err(e) = self.check_market_regime(r) {
            return RiskCheckResult::reject(e);
        }
    }
    
    Ok(...)
}
```

#### 修改建议

- 添加新检查时，**永远只增不减**（不要移除已有的检查）
- 新检查应返回 `RiskCheckResult::reject(reason)`，reason 用中文，会进入决策卡 `blockers`
- 在 [decision_engine.rs](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/signals/decision_engine.rs) 的 `decide()` 中调用，不要在 [ledger_service.rs](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/orchestration/ledger_service.rs) 中调用（执行层不应做风控决策）

```rust
fn apply_portfolio_risk_check(&mut self, ...) -> f64 {
    let mut notional = ...;
    
    // 原有 4 项检查
    let result = self.portfolio_risk.check(&portfolio);

        notional *= result.scale_factor;  // 按比例缩减

    }
    
    notional
}
```

### 4.1 DecisionEngine 源码

[backend/src/signals/decision_engine.rs](file:///d:/wan_workspase/MoneyRobert-Pro/backend/src/signals/decision_engine.rs)

核心方法 `decide()` 流程：

```rust
pub fn decide(&self, input: &DecisionInput, signal: &Signal, regime: &MarketRegime) -> DecisionOutput {
    // 1. 概率校准（v1.8）
    let p_win = match &input.calibration {
        Some(c) if c.is_reliable() => c.calibrate(signal.confidence),
        _ => signal.confidence,
    };
    
    // 2. 成本计算（v1.8 增强资金费率）
    let fee_cost_pct = ...;
    let slippage_cost_pct = ...;
    let funding_cost_pct = ...;  // 新增
    let total_cost_pct = fee_cost_pct + slippage_cost_pct + funding_cost_pct;
    
    // 3. EV 计算
    let ev = p_up * mu_up + p_down * mu_down + p_flat * mu_flat - total_cost_pct;
    
    // 4. CVaR 估算（v1.8 改进公式）
    let sigma_daily = asset_vol / 365.0_f64.sqrt();
    let holding_days = (holding_sec / 86400.0).max(1.0).min(30.0);
    let z_cvar = 2.063;
    let estimated_position_pct = 0.05;
    let cvar = z_cvar * sigma_daily * holding_days.sqrt() * estimated_position_pct;
    
    // 5. 门禁判定
    let mut blockers = vec![];
    if ev < self.config.ev_min { blockers.push("EV 不足".to_string()); }
    if cvar > self.config.risk_budget { blockers.push("CVaR 超预算".to_string()); }
    if p_win < self.config.p_min { blockers.push("概率不足".to_string()); }
    if input.data_freshness_sec > 300 { blockers.push("数据过期".to_string()); }
    
    // 6. 输出
    DecisionOutput {
        can_execute: blockers.is_empty(),
        confidence: p_win,
        ev,
        cvar,
        reasons,
        blockers,
        // ...
    }
}
```

### 4.2 修改 EV 公式

例如添加"市场冲击成本"：

```rust
// 在 decide() 中添加
let impact_cost_pct = self.compute_market_impact(
    input.notional,
    input.volume_24h,

fn compute_market_impact(&self, notional: f64, volume_24h: f64) -> f64 {
    if volume_24h <= 0.0 { return 0.0; }
    let participation_rate = notional / volume_24h;
    // 简化模型：冲击 = 0.1 × 参与率
    0.1 * participation_rate
}

let total_cost_pct = fee_cost_pct + slippage_cost_pct + funding_cost_pct + impact_cost_pct;
```

```rust
    pub fn calibrate(&self, p_raw: f64) -> f64 {
        // 优先使用 Logistic 校准（更准确）
        if let (Some(alpha), Some(beta)) = (self.logistic_alpha, self.logistic_beta) {
            let z = alpha + beta * (p_raw / (1.0 - p_raw)).ln();
            let p = 1.0 / (1.0 + (-z).exp());
            return p.clamp(0.01, 0.99);
        }
        // 回退到线性校准
        match (self.calibration_slope, self.calibration_intercept) {
            (Some(slope), Some(intercept)) => (slope * p_raw + intercept).clamp(0.01, 0.99),
            _ => p_raw,
        }
    }
}
```

```rust
pub async fn compute_vwap(&self, symbol: &str, period: Duration) -> Result<f64> {
    let klines = self.fetch_klines(symbol, period).await?;
    
    let total_value: f64 = klines.iter()
        .map(|k| (k.high + k.low + k.close) / 3.0 * k.volume)
        .sum();
    let total_volume: f64 = klines.iter().map(|k| k.volume).sum();
    
    if total_volume == 0.0 {
        return Ok(0.0);
    }
    Ok(total_value / total_volume)
}
```

### 7.1 迁移文件命名规范

`backend/migrations/NNN_描述.sql`，NNN 是三位数字编号，递增。

当前最大编号：查看 [backend/migrations/](file:///d:/wan_workspase/MoneyRobert-Pro/backend/migrations/) 目录。

### 7.2 迁移文件结构

```sql
-- 031_新增功能_描述.sql
-- 创建日期：2026-06-25
-- 作者：你的名字
-- 用途：详细说明

-- 1. DDL
ALTER TABLE xxx ADD COLUMN IF NOT EXISTS new_col VARCHAR(30);

-- 2. 索引
CREATE INDEX IF NOT EXISTS idx_xxx_new_col ON xxx(new_col);

-- 3. 注释

-- 4. 数据回填（如有）
UPDATE xxx SET new_col = 'default' WHERE new_col IS NULL;
```

### 7.3 安全迁移原则

1. **永远用 `IF NOT EXISTS`**（避免重复执行失败）：

```sql
ALTER TABLE xxx ADD COLUMN IF NOT EXISTS new_col VARCHAR(30);
CREATE INDEX IF NOT EXISTS idx_xxx ON xxx(col);
```

2. **历史数据安全转换**（参考 [030_reports_schema_fix.sql](file:///d:/wan_workspase/MoneyRobert-Pro/backend/migrations/030_reports_schema_fix.sql)）：

```sql
-- 使用 PL/pgSQL 函数捕获异常
CREATE OR REPLACE FUNCTION try_jsonb(input text) RETURNS jsonb AS $$
BEGIN
    IF input IS NULL OR input = '' THEN
        RETURN '{}'::jsonb;
    END IF;
    RETURN input::jsonb;
EXCEPTION WHEN OTHERS THEN
    RETURN jsonb_build_object('raw', input);
END;
$$ LANGUAGE plpgsql IMMUTABLE;

ALTER TABLE xxx ALTER COLUMN col TYPE JSONB USING try_jsonb(col);
DROP FUNCTION IF EXISTS try_jsonb(text);
```

3. **不删除已有列**（保持向后兼容）

4. **不修改已有列类型**（除非用 USING 安全转换）

5. **大表加列加默认值要分步**：

```sql
-- 错误（锁表）：ALTER TABLE huge_table ADD COLUMN col VARCHAR DEFAULT 'x';
-- 正确（分步）：
ALTER TABLE huge_table ADD COLUMN IF NOT EXISTS col VARCHAR;
UPDATE huge_table SET col = 'x' WHERE col IS NULL;  -- 分批
-- 最后（可选）
-- ALTER TABLE huge_table ALTER COLUMN col SET DEFAULT 'x';
```

### 7.4 迁移执行机制

迁移文件在 `db` 容器启动时通过 `/docker-entrypoint-initdb.d` 自动执行。

⚠️ **关键**：

- 首次启动：所有迁移按编号顺序执行
- 已存在的数据库重启：迁移**不会重新执行**（PostgreSQL 的 `docker-entrypoint-initdb.d` 仅在数据目录为空时执行）
- 后续新增迁移：需要手动执行

### 7.5 手动执行迁移

```rust
async fn list_items(

    )
    .bind(user.user_id)
    .bind(req.name)
    .bind(req.value)
    .fetch_one(&state.db_pool)
    .await?;
```

### 11.1 重启场景对照表

| 修改类型 | 是否需重启 | 重启方式 |

| 修改 `frontend/vite.config.ts` | ✅ 必须 | `docker compose restart frontend` |
| 修改 `.env` 文件 | ✅ 必须 | `docker compose up -d <service>` |

### 11.2 重启命令规范

⚠️ **用户规则**：Docker 镜像部署**不要在外面直接构建启动**，必须用 `docker compose`。

#### 错误方式（禁止）

```powershell
# ❌ 直接 docker build + docker run
docker build -t my-backend .
docker run -p 8001:8001 my-backend

# ❌ 直接 cargo run（生产环境）
cd backend; cargo run --release
```

#### 正确方式

```powershell
# ✅ 修改后端代码后
docker compose build backend
docker compose up -d backend

# ✅ 修改环境变量后
docker compose up -d backend

# ✅ 修改前端代码（无需重启，热更新）
# 直接保存文件即可

# ✅ 修改前端依赖后
docker compose build frontend
docker compose up -d frontend

# ✅ 全量重建
docker compose down
docker compose build
docker compose up -d
```

### 11.3 重启不丢数据

数据库和 Redis 数据存储在 Docker volume 中：

| 操作 | 是否丢数据 |
|------|----------|
| `docker compose restart` | ❌ 不丢 |
| `docker compose stop` + `start` | ❌ 不丢 |
| `docker compose down` | ❌ 不丢（仅删容器） |
| `docker compose down -v` | ✅ **丢**（删卷） |
| `docker system prune -a --volumes` | ✅ **丢** |
